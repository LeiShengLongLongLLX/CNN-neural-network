# 片上网络（NoC，Network-on-Chip）

---

## 一、概述

**片上网络（NoC）** 是一种应用于芯片（如多核处理器、SoC）内部的**通信子系统**。它借鉴了计算机网络的分层思想和拓扑结构，将计算机体系结构中的“总线”概念扩展为“网络”

**核心理念：** 将计算（资源节点）与通信（路由节点）分离，数据以**数据包**的形式在芯片上的路由网络中传输

---

## 二、关键组件

NoC 的物理实现由三大核心组件构成，它们协同工作，构成了芯片内部的“交通系统”。

| 组件 | 别名 | 功能描述 | 类比 |
| :--- | :--- | :--- | :--- |
| **网络适配器** | 网络接口 (NIC，网络接口控制器) | 作为资源节点（如CPU/GPU核心）与路由网络的**接口**。负责将核心内部的数据协议（如AXI总线）**打包**成网络可识别的数据包，并将接收到的数据包**解包**还原。 | 收发室/门卫 |
| **路由器** | 交换节点 | 根据数据包的目标地址，决定数据转发的路径。包含**缓冲区**（暂存数据）、**路由计算单元**（决策方向）、**仲裁器**（解决冲突）和**交叉开关**（物理连接输入输出）。 | 交通岗亭/交警 |
| **物理链路** | 信道 | 连接相邻路由器的**导线束**，负责实际的比特传输。通常由多条并行导线组成，可能采用差分信号等技术保证传输质量。 | 道路/高速公路 |

---

## 三、拓扑结构

拓扑结构决定了路由器与资源节点在芯片上的**物理布局**，直接影响了网络延迟、吞吐量和功耗。

### 3.1 Mesh（网格）
-   **结构**：节点排列成规整的二维网格，每个路由器只与上下左右四个邻居相连，边缘节点断开。
-   **优点**：布局规整，易于扩展，非常适合数据流高度并行的AI芯片。
-   **缺点**：中心与边缘延迟差异大，中间节点容易拥堵。
-   **应用**：Google TPU、Cerebras AI芯片。

### 3.2 Ring（环形）
-   **结构**：所有节点首尾相连，组成一个单向或双向的环。
-   **优点**：结构简单，控制逻辑简单，在节点数较少（通常<16个）时延迟极低。
-   **缺点**：扩展性差，节点增多时延迟线性增加。
-   **应用**：Intel/AMD中低端多核CPU（缓存一致性总线）。

### 3.3 Torus（环形网格）
-   **结构**：Mesh的升级版，将边缘的路由器通过**环绕链路**直接连接到对侧，形成一个环形立体结构。
-   **优点**：消除了Mesh的边界劣势，延迟更均衡，最远通信距离减半。
-   **缺点**：物理实现困难，环绕链路需要长导线，带来布线压力和时序收敛难题。
-   **应用**：超级计算机互连（如Google TPU Pod集群）。

### 3.4 3D Mesh（三维网格）
-   **结构**：在Mesh的X/Y平面基础上，增加Z轴（垂直方向）的连接。利用**硅通孔**技术实现芯片堆叠。
-   **优点**：物理距离大幅缩短，延迟和功耗显著优化，空间利用率高。
-   **缺点**：制造工艺极难，散热挑战巨大。
-   **应用**：3D堆叠缓存（如AMD 3D V-Cache），HBM内存。

### 3.5 Fat-Tree（胖树）
-   **结构**：层次化的树形结构，越靠近根节点的链路带宽越“胖”。通常分为叶脊层、枝干层和树根层。
-   **优点**：带宽不收敛，任意节点间可实现无阻塞、高性能通信。
-   **缺点**：硬件成本高，需要大量交换机和线缆，控制复杂。
-   **应用**：数据中心网络、AI服务器集群（如NVIDIA DGX SuperPOD）。

### 3.6 Butterfly（蝶形）
-   **结构**：通过多级交换网络连接所有节点，形似蝴蝶翅膀。
-   **优点**：延迟低，对分带宽高，任意节点间通常只需经过`logN`级。
-   **缺点**：布线复杂度极高，功耗大，在芯片上难以实现。
-   **应用**：传统超级计算机或特定加速器。

| 拓扑名称 | 结构描述 | 图示想象 | 核心优点 | 核心缺点 | 典型应用场景 | 可扩展性 | 硬件实现难度 |
| :---: | :---: | :---: | :--- | :--- | :--- | :---: | :---: |

| **Butterfly（蝶形）** | 多级交换网络，通过各级交换机连接所有节点。 | 蝴蝶翅膀 / 多级交换网络 | 延迟低（logN级），对分带宽高。 | 布线复杂度随节点数指数级上升，功耗大，芯片上难以实现。 | 传统超级计算机（如BBN Butterfly）、特定高速信号处理芯片。 | 中等 | 极高 |

## 四、交换技术

交换技术决定了数据在网络中如何被切割、传输和重组。它直接影响了网络的延迟和吞吐量。

### 4.1 电路交换
-   **原理**：在数据传输前，预先在源节点和目的节点之间建立一条专用的物理路径。整个传输过程中，该路径被独占，传输结束后再拆除。
-   **优点**：传输延迟低且确定，适合连续的大块数据传输。
-   **缺点**：链路利用率低，建立路径的开销大。
-   **适用场景**：对延迟要求极高的流式数据传输。

### 4.2 报文交换
-   **原理**：将整个消息作为一个整体（报文）发送。每个路由器接收到完整报文后，先存储起来，再根据目标地址转发到下一个路由器。
-   **优点**：无需预先建立路径，链路利用率高。
-   **缺点**：每个路由器都需要大容量缓冲，且存储转发延迟大。
-   **适用场景**：早期计算机网络，现代NoC中较少直接使用。

### 4.3 虚跨步交换
-   **原理**：将报文分割成多个数据包。路由器在收到一个数据包的全部内容后，才开始向下一个节点转发。
-   **优点**：相比报文交换，缓冲需求降低。
-   **缺点**：仍存在一定的存储转发延迟。
-   **适用场景**：对延迟要求不高的通用通信。

### 4.4 虫孔交换
-   **原理**：将数据包进一步分割成更小的**流量控制单元**。路由器的缓冲区只需要能存下几个FLIT即可。当头FLIT（包含路由信息）被接收并转发后，后续FLIT直接跟随头FLIT流水线式地向前流动。
-   **优点**：**延迟极低，所需缓冲小**，是目前NoC中最主流的交换技术。
-   **缺点**：如果头FLIT被阻塞，后续FLIT会阻塞在多个路由器上，形成链式阻塞。
-   **适用场景**：几乎所有现代高性能多核处理器和AI芯片的片内网络。

### 4.5 总结表

| 交换技术 | 核心原理 | 数据分割粒度 | 路径建立方式 | 延迟特性 | 缓冲区需求 | 链路利用率 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **电路交换** | 传输前预先建立专用物理路径，传输全程独占，结束后拆除。 | 无分割（连续比特流） | **需预先建立**（专用路径） | **极低**（建立后无额外延迟） | **无**（传输中无需缓冲） | **低**（独占资源） | 传输延迟确定，适合实时性要求高的流式数据。 | 链路利用率低，建立连接开销大，占用的硬件资源多。 | 音视频流传输、需要确定性延迟的场景。 |
| **报文交换** | 将整个消息作为一个整体（报文）发送，每个路由器完整接收后再转发。 | 报文（整个消息） | **无需建立**（存储转发） | **高**（每个节点都需完整存储） | **大**（需存整个报文） | **中等** | 无需预建立路径，链路共享。 | 延迟大，需要大容量缓冲，不适合片内资源紧张的环境。 | 早期计算机网络，现代NoC中极少使用。 |
| **虚跨步交换** | 将报文分割为数据包，路由器收完整个包后才开始转发。 | 数据包（Packet） | **无需建立**（存储转发） | **中等**（收完一个包再转） | **中等**（需存单个包） | **中等** | 相比报文交换，降低了缓冲需求。 | 仍存在包级别的存储转发延迟。 | 对延迟要求不高的通用通信。 |
| **虫孔交换** | 将数据包分割为更小的流量控制单元，头FLIT决定路径，后续FLIT跟随流水线传输。 | FLIT（流量控制单元） | **无需建立**（流水线式） | **极低**（流水线并行） | **极小**（只需存几个FLIT） | **高**（链路共享度高） | **延迟低，缓冲小，吞吐量高**，是当前NoC的主流。 | 头FLIT阻塞会导致后续FLIT阻塞多个路由器（链式阻塞）。 | **几乎所有现代高性能NoC**（多核CPU、AI芯片、GPU）。 |

## 五、虚拟通道技术（VC）

### 5.1 核心概念

**虚拟通道（Virtual Channel， VC）** 是一种将**一条物理通道（物理链路）逻辑上划分为多个独立的数据通道**的技术。每个虚拟通道拥有自己独立的缓冲队列，但共享同一条物理链路。

**一句话定义：** 用多个逻辑队列分时复用同一条物理线路，以解决阻塞问题。

**生活类比：**
-   **无虚拟通道**：一条单车道的公路，只有一个收费站。如果第一辆车交费慢，后面所有车都得堵着。
-   **有虚拟通道**：同一条单车道，但在收费站前设置了**多个独立的收费窗口（缓冲队列）**。虽然一次只能过一辆车，但不同方向的车可以在不同窗口排队。一个窗口堵了，其他窗口的车照样能过。

---

### 5.2 虚拟通道要解决的问题：链式阻塞

在**虫孔交换**中，一个数据包被分割成多个FLIT（流量控制单元）。当头FLIT被阻塞时，后续FLIT会停留在沿途的多个路由器中，形成一条**阻塞链**。这导致：
1.  **链路资源被无效占用**：明明物理链路是空闲的，但因为被阻塞的FLIT占据着缓冲，其他数据包无法使用。
2.  **死锁风险**：多个数据包互相等待对方释放资源，形成循环等待。

**虚拟通道的解决方案：**
通过在同一个物理端口设置**多个独立的缓冲队列**，当一个VC被阻塞时，物理链路可以切换到其他VC传输数据，从而**将阻塞限制在单个VC内部**，不影响整体链路利用率。

---

### 5.3 虚拟通道的工作原理

| 组件 | 功能描述 |
| :--- | :--- |
| **多个虚拟通道** | 在同一物理端口上，设置多个独立的缓冲队列（VC0， VC1， VC2...）。 |
| **独立缓冲** | 每个VC拥有自己专用的缓冲区（FIFO队列），用于存储属于该通道的FLIT。 |
| **共享物理链路** | 所有VC共享同一条物理导线，但在时间上**分时复用**。 |
| **仲裁机制** | 路由器中的仲裁器决定下一个时间片由哪个VC占用物理链路发送数据。 |
| **信用机制** | 下游路由器通过信用信号告知上游自己的VC缓冲区是否可用，实现流控。 |

**工作流程示例：**
1.  **分配VC**：数据包A到达路由器R1，路由器根据其目标地址或流量类别，为其分配一个空闲的虚拟通道（如VC1）。
2.  **缓冲**：数据包A的FLIT进入VC1的缓冲队列。
3.  **请求**：VC1向路由器的仲裁器申请使用物理链路发送头FLIT。
4.  **仲裁**：如果VC1申请的下一跳端口被占用，仲裁器暂时不授权VC1，但可以授权其他有数据且不阻塞的VC（如VC2）发送。
5.  **发送**：当VC1申请的下一跳端口空闲时，仲裁器在下一个时间片授权VC1发送。
6.  **信用更新**：发送成功后，上游VC1的信用数减1，下游接收后发回信用更新。

---

### 5.4 虚拟通道的优缺点

| 优点 | 缺点 |
| :--- | :--- |
| **解决链式阻塞**：大幅提升链路利用率，避免头阻塞扩散。 | **硬件开销增加**：每个VC需要独立的缓冲队列（SRAM），增加面积和功耗。 |
| **提高吞吐量**：允许更有效地共享物理链路，整体网络吞吐量可提升30%-50%。 | **控制逻辑复杂**：需要更复杂的仲裁器和信用管理机制。 |
| **避免死锁**：通过为不同类型的报文分配不同的VC，可以打破循环依赖（如将请求包和响应包分配不同VC）。 | **时序收敛难**：多VC的仲裁和切换需要在单时钟周期内完成，对设计频率有压力。 |
| **支持服务质量**：可以为高优先级数据分配专用VC，保证其延迟和带宽。 |  |
| **支持虚拟网络**：多个VC可以组合成逻辑上独立的虚拟网络，隔离不同流量类型。 |  |

---

### 5.5 虚拟通道的应用场景

| 应用场景 | 具体实现方式 |
| :--- | :--- |
| **死锁避免** | 为协议包（如请求、响应、确认）分配不同的VC，打破协议层死锁。 |
| **服务质量（QoS）** | 为实时流量（如视频、中断）分配高优先级VC，确保低延迟。 |
| **流量隔离** | 不同应用或不同核心的流量分配不同VC，互不干扰。 |
| **功耗管理** | 低负载时可关闭部分VC的时钟或电源，节省功耗。 |

## 六、片上网络服务类型

### 6.1 概述：两种服务模型

片上网络的服务质量模型主要分为两大类：**尽力服务（Best-Effort， BE）** 和**保证服务（Guaranteed Service， GS）**。这两种服务对应于不同的应用需求，类似于我们生活中的**普通快递**和**专车服务**。

| 维度 | 尽力服务（BE） | 保证服务（GS） |
| :--- | :--- | :--- |
| **核心目标** | 公平、高效、尽可能快 | 确定、可预测、有保障 |
| **延迟特性** | 可变延迟（随网络负载波动） | 有上限的延迟（worst-case保证） |
| **带宽特性** | 尽力而为，不保证 | 有保证的带宽 |
| **丢包处理** | 可能丢包（或依赖上层重传） | 无丢包保证 |
| **硬件复杂度** | 低 | 高 |
| **典型类比** | 普通快递（可能晚点） | 专车/救护车（优先通行） |

---

### 6.2 尽力服务

#### 6.2.1 定义与目标

**尽力服务**是NoC最基本的服务类型。网络会尽最大努力将数据包从源节点传送到目的节点，但**不提供任何性能保证**（如延迟上限、最小带宽）。

**核心原则：** 公平共享网络资源，尽可能提高吞吐量，降低平均延迟。

#### 6.2.2 实现机制

| 机制 | 功能描述 |
| :--- | :--- |
| **包交换** | 数据以包为单位独立传输，每个包自带路由信息。 |
| **虫孔交换** | 默认使用虫孔交换降低延迟和缓冲需求。 |
| **轮询仲裁** | 路由器中使用轮询（Round-Robin）仲裁器，保证各输入端口公平获得输出机会。 |
| **共享缓冲** | 多个数据流共享缓冲资源，按需分配。 |
| **尽力路由** | 路由算法可选择最短路径或自适应路径，以避开局部拥塞。 |

#### 6.2.3 优缺点

| 优点 | 缺点 |
| :--- | :--- |
| **硬件简单**：无需复杂的连接建立和拆除机制。 | **延迟不可预测**：网络拥塞时延迟可能剧增。 |
| **资源利用率高**：链路和缓冲可动态共享。 | **不适合实时应用**：无法满足音视频等对时序敏感的应用。 |
| **适合批量数据**：如大规模并行计算中的大部分数据流。 | **可能丢包**：缓冲满时可能丢弃数据包。 |

#### 6.2.4 适用场景

-   通用多核处理器中的大部分数据通信
-   AI芯片中非实时性的数据移动
-   缓存一致性维护流量
-   批处理任务的数据传输

---

### 6.3 保证服务

#### 6.3.1 定义与目标

**保证服务**为特定的数据流提供**有上限的延迟**和/或**有保证的带宽**。网络通过资源预留机制，确保关键流量即使在网络拥塞时也能按时到达。

**核心原则：** 确定性、可预测性、隔离性。

#### 6.3.2 实现机制

| 机制 | 功能描述 |
| :--- | :--- |
| **连接建立** | 在传输前，源节点和目的节点之间需要建立一条逻辑连接，沿途预留资源。 |
| **资源预留** | 为连接预留带宽、缓冲、虚拟通道等资源，保证这些资源不被其他流量抢占。 |
| **虚电路** | 通过虚拟通道技术，为每个GS流分配专用的虚拟通道，实现逻辑隔离。 |
| **速率控制** | 源节点按照预留的速率发送数据，避免超出网络容量。 |
| **优先级调度** | 路由器中设置优先级仲裁器，GS流始终优先于BE流。 |
| **时隙分配** | 在电路交换网络中，可分配专用的时隙给GS流。 |

#### 6.3.3 两种典型的GS实现方式

| 方式 | 原理 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **电路交换GS** | 建立专用物理路径，传输全程独占链路。 | 延迟最低且确定，适合流式数据。 | 链路利用率低，建立开销大。 |
| **虚电路GS（VC-GS）** | 通过虚拟通道预留资源，多个GS流共享物理链路但逻辑隔离。 | 链路利用率高，支持更多并发流。 | 需要复杂的资源管理和准入控制。 |

#### 6.3.4 优缺点

| 优点 | 缺点 |
| :--- | :--- |
| **延迟确定**：满足实时应用需求。 | **硬件复杂**：需要连接建立、拆除、准入控制机制。 |
| **带宽保证**：关键流量不受拥塞影响。 | **资源利用率可能降低**：预留的资源即使不用，其他流也无法抢占。 |
| **流量隔离**：不同GS流互不干扰。 | **扩展性挑战**：连接数量受限于硬件资源（如VC数量）。 |

#### 6.3.5 适用场景

-   实时音视频流传输
-   硬实时系统（如汽车自动驾驶、工业控制）
-   关键控制信号（如中断、电源管理）
-   需要确定性延迟的高性能计算通信

---

### 6.4 混合服务：BE + GS

在实际的NoC设计中，通常采用**混合服务模型**，同时支持BE和GS流量。

#### 6.4.1 典型架构：VC分离

| VC分配 | 服务类型 | 功能 |
| :--- | :--- | :--- |
| VC0 | GS高优先级 | 实时控制信号、中断 |
| VC1 | GS中优先级 | 音视频流 |
| VC2~VC3 | BE | 普通数据包 |

**工作流程：**
1.  GS流在发送前建立连接，预留VC0/VC1资源。
2.  路由器仲裁时，**严格优先级**：VC0 > VC1 > BE（VC2~VC3）。
3.  当GS流空闲时，BE流可以使用全部链路带宽。
4.  当GS流激活时，BE流自动让出预留的带宽。

#### 6.4.2 优势

-   **关键流量有保障**：满足实时需求。
-   **资源利用率高**：GS流不活跃时，BE流可借用资源。
-   **设计灵活**：可根据应用需求配置不同服务等级。

---

### 6.5 总结与对比表

| 维度 | 尽力服务（BE） | 保证服务（GS） | 混合服务 |
| :--- | :--- | :--- | :--- |
| **核心特征** | 无保证，公平共享 | 有保证，资源预留 | 关键流保证，普通流尽力 |
| **延迟** | 可变 | 有上限 | GS流确定，BE流可变 |
| **带宽** | 尽力 | 有保证 | GS流保证，BE流尽力 |
| **连接建立** | 无 | 需要 | GS流需要，BE流不需要 |
| **资源预留** | 无 | 有 | 部分预留 |
| **硬件开销** | 低 | 高 | 中等 |
| **典型应用** | 普通计算数据 | 实时音视频、控制信号 | 大部分现代SoC |

---

## 七、NoC芯片实际例子详细总结

根据你的要求，我将Intel TeraFLOPS、Eyeriss和Simba三款芯片的NoC架构详细总结如下。这三款芯片分别代表了**通用多核处理器NoC**、**领域专用加速器NoC**和**Chiplet架构NoC**三种不同的设计路径。


### 1. Intel TeraFLOPS芯片（代号Polaris，2007年）

#### 基本信息
| 维度 | 规格 |
| :--- | :--- |
| **发布年份** | 2007年（Intel Tera-Scale计算研究项目） |
| **工艺节点** | 65nm CMOS，8层铜互连  |
| **晶体管数量** | 1亿个  |
| **芯片面积** | 275 mm²  |
| **核心数量** | 80个  |
| **峰值性能** | 1.0-1.81 TFLOPS（单精度） |
| **设计功耗** | <100W（目标），实际最高230W @ 5.67GHz  |

#### NoC架构详解

**拓扑结构：10×8 2D Mesh**
- 芯片由80个Tile排列成10行×8列的二维网格 
- 每个Tile包含：**处理引擎** + **5端口虫孔交换路由器**（0.34 mm²）
- 路由器与四个方向的邻居Tile及本地的处理引擎相连 

**路由器微架构** 
- **端口配置**：5个36位端口（东、南、西、北、本地PE）
- **流水线深度**：5级流水线
- **虚拟通道**：**2个虚拟通道**（用于避免死锁）
- **缓冲**：每个虚拟通道配备16 FLIT的FIFO缓冲
- **交叉开关**：单交叉开关设计，采用**双边缘触发触发器**实现"双泵"操作（在时钟上升沿和下降沿都传输数据），相比上一代设计：
  - 交叉开关面积减少50%
  - 路由器总面积减少36%
  - 平均功耗降低13%
  - 延迟减少1个周期

**交换技术与路由** 
- **交换技术**：**虫孔交换**
- **数据包格式**：数据包分割为FLIT（流量控制单元），最小包长2 FLIT，无最大限制
- **路由算法**：确定性路由（头FLIT包含3位目的地址指示）
- **流控机制**：**背压式流控**（FIFO满时发送停止信号给上游）

**性能指标** 
- **工作频率**：标称4 GHz，最高可达5.67 GHz
- **路由器带宽**：@4GHz下，每个路由器提供**80 GB/s**通信带宽
- **网络延迟**：@4GHz下，单跳延迟**1.25 ns**
- **对分带宽**：@3.16GHz为1.6 Tbit/s，@5.67GHz为2.92 Tbit/s

**时钟架构** 
- 采用**介同步接口**（mesochronous interfaces），允许不同Tile使用同频但相位差异的时钟

**功耗管理** 
- 每个Tile划分为**21个独立睡眠区域**
- 支持动态Tile睡眠（根据工作负载）
- 最高能效：@0.6V时理论峰值**27 GFLOPS/W**

**历史意义** 
- 首个公开的80核NoC研究芯片
- 验证了2D Mesh NoC在大规模多核处理器中的可行性
- 研究成果后来融入**Xeon Phi**产品线


### 2. MIT Eyeriss芯片（2016年）

#### 基本信息
| 维度 | 规格 |
| :--- | :--- |
| **发布年份** | 2016年（MIT研究项目） |
| **应用领域** | 深度卷积神经网络加速器  |
| **PE数量** | 168个处理单元（排列为14行×12列） |
| **片上存储** | 108KB全局缓冲（GLB） |

#### NoC架构详解：领域专用的分层设计

Eyeriss的NoC设计的核心思想是：**利用CNN每层通信模式已知的先验知识，设计轻量级、可配置的专用网络，而非通用NoC** 。

**三大物理网络** 

| 网络名称 | 功能 | 数据位宽 | 拓扑结构 |
| :--- | :--- | :--- | :--- |
| **GIN（全局输入网络）** | 从GLB向PE组播**滤波器权重**和**输入特征图** | 滤波器/psum网络：**64位**；ifmap网络：**16位**  | 1条Y总线 + 12条X总线（每行一条） |
| **GON（全局输出网络）** | 从PE读取部分和并写回GLB | 64位  | 同GIN架构 |
| **LN（本地网络）** | PE之间的点对点**部分和传输** | - | 垂直方向的点对点链路  |

**逻辑拓扑** 
- 三个网络在逻辑上构成**14×12的Mesh拓扑**
- X方向链路：用于从GLB向PE发送数据
- Y方向链路：用于PE之间的本地通信

**数据流模式** 
- **滤波器权重**：沿**行方向**组播
- **输入特征图**：沿**对角线方向**组播
- **部分和**：发送给**正上方的相邻PE**

**组播机制** 
- 每个需要接收相同数据的PE在配置阶段被分配**相同的ID**
- GLB用目标PE的ID标记每条组播消息
- GIN上的交换机/控制器将数据组播给所有ID匹配的PE
- **单周期完成组播**

**流控机制** 
- 采用**开关式流控**
- GIN仅在**所有目标PE都有可用缓冲区**时才发起组播（不允许部分投递）
- GON上，GLB设置想接收的PE的ID，仅该PE允许发送psum（**无冲突，无需仲裁**）

**可配置性** 
- 网络中的交换机是**可配置且无缓冲的**
- 在每个CNN层运行前，根据该层的数据流映射配置网络
- PE可选择从psum GIN或LN获取输入部分和

**数据复用优化** 
- 采用**行驻留数据流**，同时实现三种数据复用：
  - 卷积核内复用（每个卷积核在特征图上滑动）
  - 卷积核间复用（多个输入特征图共享同一卷积核）
  - 输入特征复用（多个卷积核共享同一输入特征图）
- 访存计算比极低：AlexNet上仅**0.0029 DRAM access/multiply** 

#### 性能与功耗 
| 指标 | AlexNet | VGG-16 |
| :--- | :--- | :--- |
| **处理速度** | 35 fps | 0.7 fps |
| **功耗** | 278 mW | 236 mW |
| **Batch Size** | 4 | 3 |


### 3. NVIDIA Simba芯片（2021年）

#### 基本信息
| 维度 | 规格 |
| :--- | :--- |
| **发布年份** | 2021年（NVIDIA研究项目） |
| **架构类型** | **Chiplet多芯片模块**  |
| **Chiplet数量** | 36个小芯片  |
| **单芯片性能** | 4 TOPS  |
| **系统总性能** | 最高**128 TOPS**  |
| **能效** | 最高**6.1 TOPS/W**  |
| **应用场景** | 深度学习推理  |

#### NoC架构详解：Chiplet间的互联设计

**核心设计理念** 
- 用MCM方式集成多个**细粒度小芯片**，相比单芯片大幅降低制造成本
- 解决Chiplet间通信的开销问题

**互连架构** 
- 36个小芯片通过**封装级互连**组成MCM系统
- 支持将DNN层灵活映射到分布式计算和存储单元

**关键优化：三种Tiling优化** 
为降低Chiplet间通信开销，Simba引入三种数据局部性优化：
1. **优化1**：提升数据局部性，减少跨芯片数据传输
2. **优化2**：进一步优化
3. **优化3**：更进一步优化（具体细节原文未详述）
- 效果：相比基准层映射，**性能提升高达16%**

**性能指标** 
- **ResNet-50推理**：batch size=1时，每秒处理**1988张图像**
- **推理延迟**：**0.50 ms**

**历史定位** 
- 属于较早公开的Chiplet架构深度学习加速器研究
- 验证了用Chiplet构建大规模AI系统的可行性
- 与你之前讨论的NVIDIA MCM-GPU研究路径一致
